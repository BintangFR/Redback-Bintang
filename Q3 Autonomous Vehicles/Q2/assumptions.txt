Q2 Implementation Assumptions (Per Task)

Task 1: Task Representation
- `SimpleTask` and `ComplexTask` both expose a common processed-value interface via `ITask`, so the processor can use polymorphism without type-specific branching.
- `ITask::getProcessedValue()` returns `float` for both task types, and complex results (integer sums) are represented exactly in float for expected test ranges.

Task 2: Concurrency and Shared Queues
- The generator alternates between `SimpleTask` and `ComplexTask` using deterministic inputs to keep behavior repeatable during testing.
- A short sleep (`20 ms`) is used in loops to avoid busy-spinning and unnecessary CPU usage.
- Shutdown uses a shared `std::atomic<bool>` stop flag; processor and transmitter continue draining queues after generation stops.
- Non-blocking queue pop is used in worker loops so threads can exit cleanly once stop is requested and queues are empty.

Task 3: Data Serialization (Bitpacking)
- `TaskType` is encoded as `00` for `SimpleTask` and `01` for `ComplexTask` in bits 7..6 of byte 0; remaining bits are padding set to 0.
- Bytes 1..4 are packed in big-endian order for both float (`SimpleTask`) and `int32_t` (`ComplexTask`) payloads.
- Bytes 5..7 contain the lower 24 bits of a millisecond timestamp in big-endian order.
- For `ComplexTask`, the processed float is explicitly cast to `int32_t` at serialization time to satisfy the "float or int depending on task type" requirement.

Task 4: Unit Testing
- `main()` is excluded when `UNIT_TEST` is defined so test code can include implementation units safely.
- Bitpacking is tested with deterministic fixtures (fixed task values and timestamps), including exact byte-level assertions.
- A reusable helper is assumed for packing to avoid duplicating serialization logic across tests.

Task 5: Assumptions Documentation
- All assumptions are documented directly against the relevant task to keep traceability between implementation choices and assignment requirements.
